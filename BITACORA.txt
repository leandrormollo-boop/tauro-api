QUÉ SE HIZO
  - Modelos Pydantic en modelos/:
      cotizacion.py    → validar inputs/outputs de cotización
      producto.py      → validar productos del catálogo (HS code, etc.)
      ruta.py          → modelo de RUTAS_DEFAULT
      cliente.py       → perfil de cliente (con/sin API_KEY)
      pedido.py        → estructura de pedido + dirección + items

  - Servicios en servicios/:
      logger.py            → log_request/log_error/log_health/log_job a Sheets
      auth.py              → login mágico: generar/validar/revocar token
      rutas.py             → leer RUTAS_DEFAULT (activas + por ID)
      catalogo.py          → CRUD de PRODUCTOS_CATALOGO por cliente
      cuenta_corriente.py  → saldos y timeline de movimientos
      cotizador.py         → orquesta cotización end-to-end

  - Endpoints en endpoints/portal_cliente.py:
      GET  /portal/login          → form de email
      POST /portal/login/send     → manda link mágico
      GET  /portal/auth?token=X   → valida y crea cookie
      GET  /portal/logout         → cierra sesión
      GET  /portal/home           → saldo + pagos recientes
      GET  /portal/cotizar        → form con dropdowns de ruta
      POST /portal/cotizar        → ejecuta cotización + muestra precio
      GET  /portal/catalogo       → ver productos
      POST /portal/catalogo/add   → agregar producto

POR QUÉ ASÍ
  - Separación de responsabilidades: cada capa tiene un trabajo claro.
  - Endpoints solo orquestan — la lógica está en servicios.
  - Servicios no conocen FastAPI — se pueden testear aisladamente.
  - Modelos validan inputs antes de que lleguen a la lógica.

    que se transforme en un único archivo monstruoso.

DECISIONES IMPORTANTES
  - Sheets como base de datos (no SQL).
    Razón: Lean lo edita directo, backups automáticos de Google,
    cero infraestructura propia para mantener.

  - Login mágico por email (sin contraseñas).
    Razón: clientes B2B, simpleza, no hay "olvidé mi password".

  - Dropdowns de rutas en cotización (no escribir país/ZIP).
    Razón: evitar errores de tipeo del cliente.

  - Catálogo por cliente con HS Code pre-cargado.
    Razón: la invoice sale perfecta sin que el cliente sepa de aduana.
